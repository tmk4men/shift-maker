# 05. 現場のホンネ要件（ナレッジ）

## 出発点となった問題意識（原文ママ）

> シフト作成だが、業種によっていろんな場合があると思う。この人はいっぱい働かせてあげたいとかこの人は気に食わんからあんまり入れたくないとか。逆にこの人は新人だからいつものシフトとセットで入れないといけないけど、新人とセットで入れるような人は誰になるのかとか時間帯とか交代時間とか夜勤なのかどうかとか、そこら辺もいろいろ考えないといけないよね。あと100何万の壁みたいなのがあるけど、それを給料で超えないようにしないといけない人がいるとか、他にはいろいろ何か考えられるのかなあ。

法令や教科書的なルール（01〜04）だけでは現場のシフトは組めない。
実際に組む人の頭の中にある**「言語化されていない判断」**を、ここに機能として定義する。

---

## A. 個人ごとの「稼働の重み」（＝もっと入れたい／あまり入れたくない）

シフト作成者の頭の中には必ず「この人は多めに、この人は控えめに」がある。
これを **1つの数値（優先度スライダー）** で表現する。

```
priority: -3 〜 +3   （0 = 標準）
 +3 … 最優先で入れる（エース・生活のために稼ぎたい人）
 +1 … やや多めに
  0 … 契約どおり
 -1 … やや控えめに
 -3 … 最低限のみ（他に人がいないときだけ）
```

スコア計算では `score += -priority × 係数` として効かせる。

### 実装上の注意（重要）

- 「気に食わないから減らす」は**そのままだと法的リスク**になる。契約上の最低日数を下回る削減、
  有給取得・妊娠・育児・組合活動を理由にした削減は不利益取扱いとして違法になりうる（LAW-074, LAW-036）。
- そこでアプリ側は次の3つを**必ずセットで持つ**：
  1. `priority`（優遇度）… 自由に設定できる
  2. `契約上の最低出勤日数/時間` … priority がマイナスでも**この下限は割らない**（ハード制約）
  3. 理由メモ欄（任意・非公開）… 「スキル不足」「本人希望」など客観的根拠を残せる
- priority を下げても最低ラインは守られる、という設計にしておけば運用上安全。

### 関連する派生設定

| 設定 | 用途 |
|---|---|
| `月間の上限時間/日数` | 「入れたくない人」の実効的な制御。中立的で説明しやすい |
| `希望勤務量`（多め/普通/少なめ） | 本人の申告。priority とは別枠で持つ |
| `指名の固定枠` | 「毎週土曜はこの人」など、優遇の最も強い形 |

## B. 新人とトレーナーのペアリング

「新人は必ず誰かとセット」は**日付が同じだけでは不十分**。時間が重なっていなければ意味がない。

```
新人フラグ:      true
独り立ち予定日:  2026-08-31（この日まではペア必須）
必要ペア条件:    トレーナー資格者が、勤務時間の 70% 以上 重なっていること
                 （＝同じ勤務区分に入る、または重複時間で判定）
```

### 「誰をトレーナーにするか」のルール

トレーナー候補は次の条件をすべて満たす人：

1. `trainer: true`（教育担当フラグ）
2. 新人が入るポジションのスキルが一定以上（例：スキル3段階中2以上）
3. 新人との相性NGでない
4. その人自身が新人でない

複数いる場合の選び方（この順で優先）：

- **同じトレーナーを継続**させる（教え方のブレを防ぐ。担当トレーナーを固定できる設定を持つ）
- 直近の指導回数が少ない人（トレーナーの負担も公平に）
- 同じ勤務区分にすでに入っている人（追加コストが出ない）

### 教育に伴う副作用（見落としがちな点）

- 新人がいる時間帯は**実質の戦力が1人分下がる**ため、必要人数を +0.5〜1人 補正する設定を持つ
- 新人だけの時間帯を作らない（OPS-006）は「新人比率の上限」として実装する（例：同時間帯の50%まで）
- 独り立ち日を過ぎたら自動でペア必須を解除し、通知する

## C. 時間帯・交代時間・夜勤まわり

| 論点 | 実装 |
|---|---|
| 勤務区分（早番/中番/遅番/夜勤）の定義 | 開始・終了・休憩・日跨ぎフラグをマスタで持つ |
| 交代時の引き継ぎ | 前シフトの終了と次シフトの開始を **15〜30分重ねる**設定。重ねない場合は「引き継ぎ不能」警告 |
| 「通し勤務」 | 早番＋遅番の連結。休憩を法定どおり取れるか自動チェック（8時間超→60分） |
| 夜勤の翌日 | 「明け」を自動生成し、その翌日を休みにする（OPS-060）。明けを勤務日数に数えるかは設定 |
| 日跨ぎの労働時間帰属 | 始業日に通算（LAW-004）。週40時間判定はこの帰属で行う |
| 深夜割増の自動計算 | 22:00〜5:00の重なり時間を分単位で算出 |
| 連続夜勤・月間夜勤上限 | OPS-063 / OPS-064 |
| 夜勤を「やりたい人」 | 夜勤手当目当ての希望者がいる。夜勤の**下限**回数も設定可に（OPS-065） |
| 逆循環の禁止 | 遅番→早番、夜勤→早番などの遷移を禁止テーブルで持つ（OPS-062） |

## D. 年収の壁（扶養・社会保険）

「〇〇万円の壁」は**複数あり、しかも改正が続いている**。金額をコードに埋め込まない。

| 壁の種類 | 何が起きるか |
|---|---|
| 住民税の非課税ライン | 住民税が発生 |
| 所得税の課税ライン | 本人に所得税が発生 |
| 社会保険の適用ライン（週20時間・月8.8万円等） | 本人が社会保険に加入し、手取りが大きく減る |
| 被扶養者のライン | 扶養から外れる |
| 配偶者特別控除の逓減ライン | 世帯の手取りが減り始める |

### アプリでの扱い

```
employee.incomeCap      = 1_230_000   // 本人が超えたくない年額（自由設定）
employee.capBasis       = "年" | "月"  // 月額基準（社保の8.8万等）でも管理
employee.ytdEarnings    = 812_400     // 年初からの累計（手入力 or 実績連携）
employee.capStartMonth  = 1           // 起算月（税は1月、社保は月単位）
```

- 生成時に **「壁までの残額 → 残り勤務可能時間」** を換算し、超える割当を hard 制約で禁止
- **ペース配分**：残り月数で均等割りし、序盤に使いすぎないよう soft 制約でならす
  （例：10月時点で年枠の90%を使っていたら、11〜12月がほぼ入れなくなる＝現場が回らない）
- 深夜割増・残業割増で**想定外に賃金が伸びる**ため、割増込みで見積もる
- 交通費は原則非課税だが社保の被扶養判定では含む場合がある → 「壁の計算に交通費を含める」設定
- 年末（11〜12月）に自動で警告：「このペースだと 12/20 に上限到達、以降シフト不可」
- 逆に **「壁ギリギリまで働きたい」** ニーズもある。上限に近づける／余らせないモードを持つ

## E. その他、現場で実際に効いてくる要素（追加アイデア）

### 人に関するもの
- **相性NG**：一緒に組ませるとトラブルになる組み合わせ。理由は非公開で保持
- **相性GOOD**：連携が良く回るペア。繁忙日に寄せる
- **カップル・親族**：同時勤務を避ける（内部統制／私語）または合わせる
- **口の堅さ・現金取扱の可否**：レジ締めや金庫を任せられるか
- **クレーム対応可否**：責任者不在時に対応できる人
- **モチベーション／定着リスク**：希望が通らない状態が続くと辞める。連続却下回数を監視して警告
- **ベテランの休みが重なる日**を作らない（技術力の総和に下限を設ける）
- **キーマン依存の可視化**：その人が抜けたら回らない日を検出

### 時間・生活に関するもの
- **保育園のお迎え時刻**＝終業のハード上限
- **終電・始発**：深夜勤務者の帰宅可否、タクシー代の予算計上
- **通学・試験期間**：学生は試験2週間前から不可、など期間指定
- **ダブルワーク先のシフト**：重複禁止＋移動時間確保＋労働時間通算（LAW-008）
- **通勤距離**：早番（早朝）は近隣者を優先
- **本人の体調・持病**：連勤上限を個別に下げる

### 店舗・業務に関するもの
- **曜日別・時間帯別の売上/客数予測**から必要人時を逆算
- **納品日・棚卸日・締め日**は特定の人員が必要
- **イベント・天候（雨で客数減）**による当日調整
- **新人研修・全体ミーティング**の時間を勤務として組む
- **複数店舗のヘルプ**：移動時間、応援可否、応援手当
- **繁忙期の採用計画**：不足が続くスロットを「採用が必要な時間帯」として出力

### 運用に関するもの
- **希望提出の締切／確定公開日**の管理（LAW-072）
- **確定後の交代申請フロー**（本人同士でスワップ → 承認 → 再検証）
- **当日欠勤時の代替候補提示**（制約を満たす人を残業余力順に）
- **公平性の履歴繰越**（今月損した人が来月優遇される）
- **「なぜこの人なのか」の説明表示**（納得感がクレームを防ぐ最大の要素）
- **前月/翌月との連続性**（月跨ぎの連勤・夜勤明け）
- **シフトの複製**（先月をベースに作る／固定シフトのテンプレ化）

## F. このアプリの設計方針

1. **AIを使わない。** 決定的なルールベース（スコアリング＋局所探索）。同じ入力なら同じ出力。
2. **ローカル完結。** サーバ送信なし。データはブラウザの localStorage ＋ JSONの書き出し/読み込み。
3. **操作は簡単に。** 最低限「従業員」「勤務区分」「必要人数」「希望休」の4つを入れれば作れる。
   細かい制約はすべて既定値つきで、触らなくても動く。
4. **説明できる。** 生成結果の1マスごとに、なぜその人になったかを表示する。
5. **法令は絶対に破らない。** LAW-* はハード制約固定で、UIから無効化できない。
