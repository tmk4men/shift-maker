# 04. 制約のモデル化（実装仕様）

ルールを「日本語の箇条書き」のままではプログラムにできないので、
**すべての制約を同じ形の構造体**に落とし込む。

## 1. 制約の共通スキーマ

```js
{
  id: "OPS-036",            // ルールID（01〜03のIDと対応）
  name: "最大連勤日数",
  scope: "person",          // person | day | slot | global
  type: "hard",             // hard | soft（設定で変更可）
  weight: 100,              // soft のときのペナルティ点（大きいほど守りたい）
  params: { maxConsecutive: 5 },
  enabled: true,
}
```

- **hard** … 1件でも破ったら「解なし」。生成器は絶対に割り当てない。
- **soft** … 破ってよいが `weight × 違反量` を減点。合計減点の最小化を目指す。

## 2. データモデル（最小構成）

```
Employee   : id, 氏名, 雇用区分, 時給, 役割[], 資格[], スキル{ポジション:0-3},
             新人フラグ, トレーナーフラグ, 夜勤可否, 未成年フラグ,
             契約(週所定日数/時間, 最低・最大日数, 最大連勤),
             アベイラビリティ(曜日×時間帯), NG相手[], 推奨ペア[],
             年収上限(壁), 年初からの累計賃金, 希望勤務量(多め/普通/少なめ)
ShiftType  : id, 名称(早番/遅番/夜勤), 開始, 終了(翌日フラグ), 休憩分,
             深夜跨ぎ, 必要スキル, 色
Demand     : 日付 or 曜日 × ShiftType × { 必要人数min/max, 必須役割[] }
Request    : 社員 × 日付 × (希望休/絶対休/出勤希望/有給)
Assignment : 日付 × ShiftType × 社員   ← 生成結果
```

## 3. 解き方（AIを使わない決定的アルゴリズム）

外部APIも機械学習も使わず、**ルールブック駆動の探索**で解く。

```
Step 0  前処理
        前月末データ読込（連勤・インターバル・夜勤明けの継続判定用）
        各社員の残枠計算（残日数・残時間・壁までの残賃金）

Step 1  確定枠の反映
        絶対休・有給・休職・研修などを先に埋める（動かせない）

Step 2  希少人材から埋める（最も制約の強いスロットから）
        必須役割（責任者・有資格者）→ 夜勤 → 土日祝 → 平日
        ＝ Most Constrained Variable ヒューリスティック

Step 3  各スロットについて候補者をスコアリング
        hard 制約を1つでも破る人は候補から除外
        soft 制約のスコア合計が最良の人を割り当てる
        （同点なら「累積不公平度が大きい人」を優先＝公平性の自動補正）

Step 4  未充足スロットが残ったら修復（局所探索）
        ・他日の同一人物の割当と入れ替え（swap）
        ・別の人へ移す（move）
        改善しなくなるまで一定回数繰り返す

Step 5  検証
        全 hard 制約を再チェック → 違反0を保証
        soft 違反は一覧化して「なぜこうなったか」を提示

Step 6  未充足が残る場合
        不足しているスロットと、その原因になった制約を明示し、
        「どの制約をどれだけ緩めれば解けるか」を提案する
```

同じ入力なら**必ず同じ結果**になるよう、乱数は使うとしても固定シード。

## 4. 候補者スコア（Step 3の中身）

```
score = Σ ( weight_i × violation_i )   … 小さいほど良い
```

| 項目 | 向き |
|---|---|
| 希望休に当たる | 大きく加点（避ける） |
| 出勤希望に当たる | 減点（優先） |
| 契約日数・時間に対する充足の不足 | 不足しているほど優先 |
| 累積の不公平度（夜勤・土日・希望却下の履歴） | 損している人を優先 |
| スキル適合度 | 高いほど優先 |
| 人件費（時給×時間＋割増） | 予算逼迫時のみ効かせる |
| 年収の壁までの残り | 残りが少ない人は避ける |
| 個別の優遇度／稼働上限（05参照） | 設定に従う |

## 5. 緩和順序（解が見つからないときに何から諦めるか）

上から順に緩める。**法令（LAW-***）は絶対に緩めない。**

1. 公平性（偏りの許容幅を広げる）
2. 希望休（「できれば休」のみ。絶対休は不可）
3. 経験者と新人のバランス
4. 上限人数（少し多めの配置を許す）
5. 予算（超過を許容し警告）
6. 最大出勤日数（本人合意が前提）
7. → それでも解けなければ「人員不足」として採用/ヘルプ要請を提案

## 6. 説明可能性（AIを使わないからこそ強い部分）

生成結果には必ず根拠を付ける。

- 「7/15 の夜勤に佐藤さんを入れた理由：夜勤可能者3名のうち、田中さんは前日夜勤で連続上限、鈴木さんは希望休。佐藤さんは今月の夜勤回数が最少（2回）」
- 「山田さんをこれ以上入れられない理由：年収上限まで残り 42,000円（＝残り約38時間）」

このトレースをUIに出せることを実装の必須要件とする。
